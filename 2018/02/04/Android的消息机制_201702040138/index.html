Android的消息机制

发表于 2018-02-04 | 分类于 Android

Android的消息机制概述

  1. Android的消息机制主要指Handler的运行机制。通过Handler我们可以轻松得将一个任务切换到Handler所在的线程中去执行。而Handler的运行，需要底层的MessageQueue和Looper的支持。

  2. MessageQueue，消息队列。他的存储了一组消息，以队列的形式提供插入和删除工作。其内部结构是采用单链表的数据结构实现的队列。

  3. Looper，消息循环。他会无限循环得去查询是否有新的消息，有则处理，否则等待。

  4. ThreadLocal，线程本地。期作用是可以在不同的线程中互不干扰地存储和获取数据结构。而Looper则是利用ThreadLocal独立得存储在不同的线程中的。

    *注意：线程是默认没有Looper实例的，而主线程（即UI线程）在被创建时初始化了Looper，这就是我们在主线程中可以使用Handler的原因。

  5. Android是不允许在除主线程之外的其他线程中更新UI的，这是因为Android的UI控件不是线程安全的，如果在多线程中并发得访问UI控件，会使控件处于不可预料的状态。如果在主线程之前的其他线程中操作UI，则程序会抛出 CalledFroWrongThreadException 异常。ViewRootImpl对UI操作做了线程验证，如下：

  而Android又建议我们不要在主线程中进行耗时的操作，否则会导致程序ANR（application no response）。所以Handler的作用在解决子线程中无法访问UI的矛盾上十分显著。

  1. Handler的创建是要依赖所在线程中的Looper实例的，Handler创建时，会采用当前线程中的Looper来构建内部的消息循环系统，如果不存在则会抛出异常。所以在主线程（主线程默认初始化了Looper实例）之外的其他线程使用Handler时必须为该线程创建Looper。

  2. Handler的工作过程如下：

  Handler创建时，其所在线程必须要有Looper实例。Handler创建后就可以和线程内部的Looper以及MessageQueue一起协同工作了。Handler负责发送消息，MessageQueue负责存储消息，Looper负责在Handler所在线程中处理消息（利用ThreadLocal可以在不同的线程中存储不同的Looper实例，这能保证Handler发送的消息必定是由Handler所在的线程中的Looper实例处理）
  详细过程：Handler通过post方法将一个Runnable投递到Handler内部的Looper中去处理，也可以通过send方法发送一个消息到MessageQueue中由Looper处理。其实post方法最终也是通过send方法完成的。接下来主要看send方法，Handler的send方法被调用时，他会调用MessageQueue的enqueueMessage方法将消息置入队列中，然后Looper发现有新消息之后就会处理这个消息，最终被投递至Looper的Runnable或者Handler的handleMessage方法会被调用。
